Single Sign-On mit Azure AD und Google Workspace

Diese Seite beschreibt, wie Sie Single Sign-On für DMH SME Demo Cloud aktivieren. Die Konfiguration besteht aus drei Phasen: Vorbereitung im Identity-Provider, Eintragen der Metadaten im DMH-Adminportal, und ein Testlauf mit einem dedizierten Test-Benutzer.

Unterstützte Identity Provider

  • Microsoft Entra ID (vormals Azure AD) — Tarif Business oder höher erforderlich
  • Google Workspace — alle Tarife
  • Okta — Tarif Enterprise erforderlich

Schritte für Microsoft Entra ID (Azure AD)

  1. Im DMH-Adminportal unter /admin/sso die Option SAML 2.0 auswählen und auf Neue Verbindung anlegen klicken. Notieren Sie die generierte ACS-URL und den Audience-Identifier urn:dmh-sme-demo:saml. Sie brauchen beide im nächsten Schritt.
  2. Im Entra-Admin-Center unter Unternehmensanwendungen → Neue Anwendung → Eigene Anwendung erstellen eine Anwendung mit dem Namen "DMH SME Demo Cloud" anlegen und SAML 2.0 als Single-Sign-On-Methode auswählen.
  3. ACS-URL und Audience-Identifier aus Schritt 1 in die Entra-Anwendung einfügen. Anschließend das Verbund-Metadaten-XML herunterladen — dieses XML enthält das Signaturzertifikat, das DMH zum Validieren der SAML-Assertions benötigt.
  4. Im DMH-Adminportal das Metadaten-XML hochladen und die Verbindung aktivieren. Standardmäßig werden die Attribute email, name und group_id erwartet. Andere Attribut-Namen lassen sich im Reiter Attribute-Mapping umbiegen.
  5. Mit einem Testbenutzer aus Entra einloggen und prüfen, dass die Rolle korrekt zugeordnet wird. Verwenden Sie hierfür idealerweise einen extra für SSO-Tests angelegten Benutzer mit eindeutiger Gruppenzugehörigkeit.

Schritte für Google Workspace

Google Workspace verwendet ebenfalls SAML 2.0. Im Google-Admin-Konsolen-Bereich Apps → Web- und mobile Apps → Benutzerdefinierte SAML-App eine neue App anlegen, die ACS-URL und Audience-Identifier aus DMH einfügen, die SAML-Metadaten herunterladen und in DMH hochladen.

Just-In-Time-Provisioning

Per Default wird ein Benutzer beim ersten Login automatisch in DMH SME Demo Cloud angelegt (JIT-Provisioning), wenn das Attribut group_id einer in DMH konfigurierten Rolle entspricht. Diese Funktion lässt sich unter /admin/sso → Provisioning abschalten, falls Sie stattdessen SCIM bevorzugen.

Fehlerdiagnose

Schlägt der SAML-Login fehl, finden Sie das Trace-Log unter /admin/sso/diagnostics. Häufigste Ursache: Clock-Skew zwischen IdP und DMH-Server größer als 300 Sekunden — die SAML-Assertion gilt dann als ungültig. Synchronisieren Sie die Uhren beider Systeme via NTP. Zweithäufigste Ursache: ein veraltetes Signaturzertifikat, das nach einer IdP-Rotation nicht mehr im DMH-Adminportal aktualisiert wurde.
